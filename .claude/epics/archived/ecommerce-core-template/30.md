# Task: Model Re-exports & Alembic Migration

## Description

Wire up `ecommerce_core/models/__init__.py` to re-export all 16 models, then generate the single Alembic initial migration that creates all tables. Validate that `Base.metadata.create_all()` output matches the migration with zero drift.

## Acceptance Criteria
- [ ] `ecommerce_core/models/__init__.py` imports and re-exports all models: `ProductCategory`, `ProductTemplate`, `ProductVariant`, `ProductAttribute`, `ProductAttributeValue`, `ProductVariantAttributeLine`, `Partner`, `Address`, `Pricelist`, `PricelistItem`, `Coupon`, `Cart`, `CartLine`, `SaleOrder`, `SaleOrderLine`, `ShippingMethod`, `PaymentTransaction`
- [ ] `python -c "from ecommerce_core.models import *"` exits with no errors
- [ ] No circular import errors (run: `python -c "import ecommerce_core.models"` and verify)
- [ ] `alembic/versions/0001_initial.py` generated (or hand-written) to create all tables in correct dependency order
- [ ] `alembic upgrade head` on a fresh SQLite DB creates all tables without error
- [ ] Table count matches model count (17 tables: 16 models + 1 association table `product_variant_attribute_line`)
- [ ] `alembic downgrade base` drops all tables cleanly

## Technical Details

Import order in `__init__.py` must respect FK dependencies to avoid mapper configuration errors:
1. `category.py` (no FKs to other models)
2. `attribute.py` (no FKs except self)
3. `product.py` (FKs: Category, Attribute)
4. `partner.py` (no FK deps)
5. `pricelist.py` (no FK deps beyond Base)
6. `coupon.py` (no FK deps)
7. `shipping.py` (no FK deps)
8. `cart.py` (FKs: Partner, Pricelist, Coupon, ShippingMethod, ProductVariant)
9. `order.py` (FKs: Partner, Address, ShippingMethod, Coupon, ProductVariant)
10. `payment.py` (FK: SaleOrder)

For `0001_initial.py`: use `op.create_table()` calls in the same dependency order. Do not rely on `autogenerate` alone — verify the output manually.
